1. 面试问题
1.1 前端数据规范是怎么制定的  
大厂的埋点规范（通常叫 DRD - 数据需求文档），规定了前端传给后端的 JSON 必须是强结构的。一般分为两层：
- 外层（Common 公共参数）： 解决 Who, When, Where。
  - device_id (设备指纹，未登录用户的唯一凭证)
  - uid (用户ID，登录后才有)
  - session_id (会话ID，比如用户打开APP到退出算一个session，用来做链路串联)
  - os, app_version, network, timestamp
- 内层（Properties 业务属性参数）： 解决 What, How。
  - event_name (事件名，必须标准化，如 view_page, add_to_cart, pay_order)
  - item_id, category, price, page_source (具体业务字段)



1.2 整条链路怎么统计？根据什么标准？
要证明用户是“顺理成章”完成这一套动作的，仅仅靠同一个 ID 是不够的（因为他可能是上个月搜索的，今天才买，这不算一次完整的链路转化）。
- 标准一：Session ID (会话标识) 前端每次打开 App，生成一个生命周期为 30 分钟的 session_id。只要在这个时间内的所有操作，都算同一条链路。
- 标准二：时间窗口内的有序事件 (漏斗逻辑) 大厂底层怎么查漏斗？比如查询“今天，搜索 -> 查看商品 -> 加购 -> 支付”的转化率。 在 MySQL 里写极其痛苦（需要多层嵌套和表关联）。但在 ClickHouse 中，有一个专门的函数叫 windowFunnel（漏斗聚合函数）。你只需规定一个时间窗口（比如 1 小时内），它会自动按顺序帮你算出每一步留下了多少人。
典型的**全链路漏斗分析（Funnel Analysis）**场景。要串联起用户的整条转化链路，仅仅依赖用户 ID 和商品 ID 是不够的。
我们统计链路的标准核心有两个：一是唯一设备指纹（Device ID），二是会话生命周期（Session ID）。
具体统计逻辑如下： 首先，在前端埋点时，只要用户打开应用，就会生成一个 session_id。不论用户登录与否，他从‘搜索 -> 点击 -> 加购 -> 结算’的每一步埋点 JSON 中，都会携带同一个 device_id 和 session_id。
当数据落到我们的分析库（比如 ClickHouse）后，我们并不是用传统的 SQL 去死磕多表关联。而是基于 device_id 分组，使用 ClickHouse 原生的 windowFunnel（时间窗口漏斗函数）。 我们设定一个时间窗口（比如 2 小时），以 event_name 为条件，依次设定 search -> view -> add_cart -> pay 作为漏斗层级。底层引擎会自动在这个时间窗口内，根据时间戳的先后顺序，匹配出一条完整的消费链路，并算出每一步的转化率。这才是电商全链路归因和统计的标准做法。”



1.3 做报表用什么工具？PV、UV、GMV 怎么算
“报表展示这块，如果是常规的管理后台，我们后端会提供按天/按小时聚合的 API，前端配合 ECharts 进行可视化渲染；如果是复杂的运营分析大盘，我们直接将底层数据库（如 ClickHouse）接入了 Metabase / Superset 这种成熟的 BI 商业智能工具，让运营自己拖拽生成看板。
关于核心指标的统计逻辑： PV 就是统计特定页面的事件触发总数； GMV 是统计订单表里的 order_amount 总和，口径上包含已支付和未支付的下单金额； 最核心的是 UV，我们统计的口径是基于 device_id 进行去重（COUNT DISTINCT）。但考虑到大促期间数据量庞大，精确去重极其消耗内存，所以在底层大数据架构上，我们会利用 Redis 或 ClickHouse 的 HyperLogLog (HLL) 函数进行估算，在极低内存消耗下做到 99% 以上的精确度，完美支撑大盘实时 UV 的展现



1.4 一天的数据1pb存在数据库里面怎么存
- MySQL 的宿命（OLTP 行式存储）： MySQL 的底层结构是 B+ 树，它是按“行”来存数据的。它的设计初衷是**“记账”**（比如扣减库存、记录一笔订单），这要求极高的事务安全性（ACID）。当你用它去统计“今天有几千万人看了某条裙子”时，它必须一行一行地去读磁盘，不仅极其缓慢，而且会把数据库的 CPU 和 I/O 全部占满，导致真正的购物订单进不来，系统直接崩溃。
- 大厂的解法（OLAP 列式存储）： 面对海量报表和聚合分析，大厂会引入像 ClickHouse 或 Apache Doris 这样的“列式数据库”。它们是按“列”存数据的。当你要计算“总曝光量”时，它不需要读其他无关的字段，直接把“曝光量”这一列的数字全部拉出来，配合底层极其残暴的 CPU 向量化计算，PB 级的数据 COUNT() 或 SUM() 也能做到毫秒级响应。
1.4.1 大厂标准架构流转链路
一条曝光埋点数据产生后，它的旅程是这样的：
1. 产生： 前端页面滑过商品，瞬间产生上万条曝光记录（JSON 格式）。
2. 削峰（缓冲层）： 这些记录发送给后端网关后，绝对不写 MySQL，而是直接扔进 Kafka（消息队列） 里。Kafka 就像一个无底洞，写盘速度极快，能轻松抗住双十一的洪峰。
3. 清洗（计算层）： 后台运行着 Apache Flink（流处理引擎），它像抽水机一样从 Kafka 里抽数据，进行实时清洗、过滤掉作弊流量。
4. 落盘（存储层）： Flink 洗完数据后，批量写入 ClickHouse / Doris。运营的报表系统直接连 ClickHouse 查数据。
1.4.3 面试对线话术（建议熟读并转化为自己的语言）
“欧阳老师，如果我们要抗住黑五或者双十一 PB 级别的曝光数据洪峰，传统的 MySQL + Redis 架构在物理层面上是绝对吃不消的。
这种场景下，大厂的标准解法是进行业务库与分析库的物理隔离（读写分离演进版）。 我们的核心交易（如下单、支付）依然走 MySQL 保底，保证强一致性。而对于海量的曝光和点击埋点，我的架构设计是：
1. 数据通过网关后直接打入 Kafka 集群进行削峰填谷，彻底将高并发流量与业务库隔离。
2. 下游使用 Flink 等流处理引擎消费 Kafka，做实时的清洗和聚合。
3. 最终清洗好的数据，写入专门为海量查询而生的列式数据库，比如 ClickHouse。利用它的列式存储和向量化引擎，PB 级数据的多维报表统计也能做到秒级甚至毫秒级出结果。这就彻底解决了大数据量下的查询瓶颈。”
1.4.4 CK
1.4.4.1 稀疏索引 (Sparse Index)
- MySQL 的 B+ 树是稠密索引，一行数据对应一个索引节点，上亿数据时 B+ 树深不可测。
- 深层原理（你要说的）： ClickHouse 的核心引擎 MergeTree 使用的是稀疏索引。它默认每隔 8192 行数据才提取一个主键（Primary Key）作为索引条目。这 8192 行被称为一个 Granule（颗粒）。在查询时，CK 不是去找具体的哪一行，而是快速定位到可能包含目标数据的 Granule，然后把这一大块数据直接扔进内存里扫描。这极大地减小了索引的体积，使得索引完全可以放进内存中。
1.4.4.2 CPU 向量化执行 (SIMD)
- 深层原理（你要说的）： 这才是 CK 快的物理本质。传统的数据库（如 MySQL）是按行读取，执行模型是 Volcano Model（火山模型），一行一行地调用函数去处理，函数调用的开销极大。而 CK 是按列读取的，它利用了现代 CPU 的 SIMD（单指令流多数据流，如 SSE4.2、AVX-512 指令集）。当做 SUM() 或 COUNT() 时，CK 会将一整批数据（比如几千个整型数字）加载到 CPU 的超宽寄存器中，在同一个 CPU 时钟周期内，用一条机器指令同时对这一批数字完成加法运算。这是物理层面的降维打击。
1.4.4.3 LSM-Tree 的变种写入机制
- 深层原理（你要说的）： 面对于 PB 级的并发写入，CK 底层没有像 MySQL 那样去维护复杂的 B+ 树分裂和页合并。写入时，它直接在磁盘上追加生成小的数据片段（Data Part）。后台会通过 Merge 线程，不断地将这些小的 Part 归并合并成大的 Part。这种彻底的**顺序写（Sequential Write）**榨干了磁盘的 I/O 极限，所以单节点就能抗住极高的写入 QPS
1.5 es数据库
1.5.1 核心原理解析：MySQL LIKE vs ES 倒排索引
- MySQL 的正排索引（Forward Index）： 就像一本书的目录。你知道文章在第几页，翻过去看内容。但如果你要在全库找“连衣裙”这个词，即使加了索引，遇到 LIKE '%连衣裙%' 时，MySQL 也只能一页一页（全表）去翻，数据一过百万，直接卡死。
- ES 的倒排索引（Inverted Index）： ES 的做法是把书的最后几页的“索引表”拿过来。它先用分词器（Analyzer）把商品标题切碎。比如“夏季修身连衣裙”，切成“夏季”、“修身”、“连衣裙”。然后它建立一个大字典，记录每一个词分别出现在哪些商品 ID 中。
深度一：ES 搜索底层的“硬核魔法” (Lucene 源码级解析)
之前我说 ES 是用倒排索引，包含字典（Term Dictionary）和倒排表（Posting List）。面试官如果稍微资深一点，马上会反问：“几十万的 SKU，分词之后词典极大，内存根本放不下，磁盘 I/O 又慢，ES 底层到底是怎么解决查询性能的？”
这时候，你不能再泛泛而谈，必须甩出下面这三大核心底层技术：
1.5.2 词典索引：FST (Finite State Transducer)
- 浅层认知： 用 Hash 表或者 B+ 树存词典。
- 深层原理（你要说的）： ES 底层的 Lucene 引擎并没有把整个词典放进内存，而是为词典构建了一个叫做 Term Index（词典索引）。这个索引的数据结构是 FST（有限状态转换器）。FST 不仅能像 Trie（字典树）一样共享前缀，还能共享后缀！它的内存占用极小，可以将原本需要几个 G 的词典索引压缩到几十 MB，完全常驻物理内存。每次查询时，先在内存中通过 FST 复杂度为 $O(len(str))$ 极速定位到磁盘上 Term Dictionary 的 Block 偏移量，极大减少了磁盘随机寻道次数。
1.5.2 倒排表压缩：FOR (Frame of Reference) 与 Delta Encoding
- 浅层认知： 倒排表就是一个存着文档 ID 的数组，比如 [73, 300, 302, 332, 343, 372]。
- 深层原理（你要说的）： 如果搜索“女装”，可能会命中上千万个商品 ID。如果每个 ID 用 32 位整型存，上千万 ID 会瞬间打爆 CPU 缓存和内存。Lucene 采用 Delta 差值编码，不存原始 ID，只存与前一个 ID 的差值，上面的数组变成了 [73, 227, 2, 30, 11, 29]。发现了吗？数字变小了！然后 Lucene 使用 FOR (Frame of Reference) 算法，按每 256 个文档打成一个 Block，计算出这个 Block 里最大的数需要几 bit（比如最大数是 227，只需要 8 bit），那么这整个 Block 里的所有数字都用 8 bit 强行打包，摒弃传统的 32 bit。这直接把倒排表的体积压缩了 70% 以上，极大地提升了 CPU L1/L2 Cache 的命中率。
1.5.3 多条件联合查询：Roaring Bitmaps (咆哮位图)
- 浅层认知： 多条件查询（比如：夏季 AND 连衣裙）就是把两个倒排表的数组拿出来求交集。
- 深层原理（你要说的）： 当过滤条件非常复杂时，ES 会把查询结果缓存起来（Filter Cache）。但它不是存普通的数组，而是使用了 Roaring Bitmap 数据结构。对于稀疏的数据，它用数组存；对于密集的数据，它转成 Bitmap（位图）存。在做 AND 或 OR 的联合检索时，直接在 CPU 层面执行位运算，性能比传统的跳表（Skip List）求交集快了几个数量级。

1.6 埋点表字段的“无感扩展” 
1.6.1 核心原理解析：为什么不能随便 ALTER TABLE？
- 浅层认知： 业务要加个“停留时长”字段，写句 ALTER TABLE ADD COLUMN 就行了。
- 深层原理（你要说的）： 在千万甚至亿级别的 MySQL 埋点表里执行 DDL 操作，会引发元数据锁（MDL, Metadata Lock）。即便 MySQL 5.6 之后支持了 Online DDL，在修改期间也会产生大量的 CPU 和 I/O 消耗，甚至主从延迟。在大厂，大表的 DDL 必须走复杂的工单，还要用 gh-ost 或 pt-online-schema-change 这种影子表工具，极其痛苦，绝不是“随便加字段”的。
1.6.2 大厂解法一：MySQL 的 JSON + 虚拟列 (Virtual Columns)
- 深层原理（你要说的）： 为了彻底解耦业务和表结构，我们可以在 MySQL 中设计一个 ext_info 的 JSON 字段。更硬核的是，如果运营以后非要拿 JSON 里的某个新属性（比如 stay_time）来做条件查询，直接查 JSON 会导致全表扫描。此时，你可以利用 MySQL 5.7+ 引入的 Generated Columns（生成列/虚拟列） 技术，为 JSON 里的 stay_time 创建一个虚拟列，并在这个虚拟列上建立 B+ 树索引。这样既实现了无感加字段，又保证了新字段的查询性能！
1.6.3 大厂解法二：拥抱 Schema-free (HBase / MongoDB)
- 深层原理（你要说的）： 如果埋点本身就是非结构化的，最好的防守就是换数据库。比如使用宽表数据库 HBase，它基于列族（Column Family），列是可以动态无限扩展的；或者使用 MongoDB（BSON 文档），没有 Schema 限制，你传什么字段它存什么字段。

1.7 未登录用户的全链路追踪 (OneID 与 ID-Mapping)
这是面试官（张明）对你实施的最后一次降维打击。他问的是“漏斗转化”，你答的是“没登录不统计”。我们要把这个局彻底翻过来，抛出大厂数据中台最核心的概念之一：ID-Mapping（账号统一）。
1.7.1 核心原理解析：割裂的孤岛与连通图
- 业务痛点： 用户第一天用手机浏览器看了商品（匿名），第二天下载 APP 看了商品（匿名），第三天注册账号并下单（实名）。怎么证明这是同一个人？
- 深层原理（你要说的）： 1. 端侧设备指纹： 前端必须有一套生成唯一 Device_ID（设备指纹）的机制（比如基于浏览器 Canvas 指纹、或者 App 的设备机器码）。任何匿名行为，全部挂在这个 Device_ID 下。 2. 服务端双绑： 核心转折点在于**“登录/注册”那一瞬间**。前端发出的登录请求，必须同时带上 Device_ID 和新的 User_ID。 3. 数仓层 ID-Mapping (OneID)： 数据进了离线数仓（比如 Hive）后，每天晚上跑批处理脚本。利用图计算（如 Spark GraphX）中的连通子图算法，把同一个 Device_ID 过去所有的匿名轨迹，强行并入这个 User_ID 的名下，实现数据的“认祖归宗”。
2. 整体设计
2.1 数据采集方式
数据采集是实时还是离线的？采集时是否会丢失数据？

求职者回应
大部分是准实时的，比如用户行为数据是前端埋点 POST 过来的，销售数据是从订单系统同步过来的。偶尔会有延迟，但基本没出现数据丢失，因为做了兜底方案。

2.2 兜底方案细节
兜底方案具体如何实现？

求职者详细说明兜底方案
用户行为数据兜底：
前端埋点请求失败时，最多重试三次，间隔 1 秒。
若重试失败，数据存本地缓存（local storage），网络恢复后自动重发。
后端接口接收数据时，先写本地日志记录请求参数和时间。
若写入数据库失败，定时任务每十分钟扫描日志，对比数据库，缺失则重新写入。
销售数据兜底：
订单系统同步数据时，通过消息队列（RabbitMQ）发送消息。
若消息未收到或消费失败，每五分钟定时拉取订单系统接口未同步的数据。
每日凌晨全量校验，自动补偿缺失数据。

话术
我就具体说一下，这个兜底我记得是做了两个方面的动作，因为用户行为和销售数据来源不一样，分开兜底才稳妥，像之前也踩过一些坑，比如刚开始是没有把这个分开做，就会出现数据丢失的一些情况，后面就做了一些优化。
第一、用户行为数据是前端埋点 POST 请求过来的，主要怕的是网络波动，比如前端发了之后，后端没收到，或者后端收到了之后，写入数据库失败了。所以前端那边就让他们做了一些重试机制，比如最多重试三次，每次间隔一秒钟，如果还是失败，就把这条数据存到本地缓存 local storage 里面，等用户网络恢复正常之后，下次再打开页面的时候，再自动地重新进行发送，不会把它直接丢弃。
在后端这边接收请求的接口也做了一些兜底，比如就算前端没有进行重试，后端在接收数据的时候，会先写一份本地日志，把请求参数、接收的时间都记录下来。如果写入数据库失败了，比如数据库连接超时，像这种情况，还会有一个定时任务，比如每十分钟去扫描一次本地日志，对比数据库里面的数据，如果发现日志里面有，而数据库里面没有，就再做重新写入，相当于在后端再补一道。
第二、销售数据从订单系统同步过来，怕的是订单系统那边的接口崩掉了，或者消息同步也丢掉了，做的就是消息通知加定时拉取，像这种双重兜底。具体的说就是订单系统那边如果有新的销售数据，会先发一条消息到 rocketmq 当中，消费消息同步的数据，
如果消息没有收到，或者消费失败了，还会有一个五分钟的定时，去订单系统的接口拉取一次未同步的销售数据，拉取的时候还要回传一个最新同步的订单 id，订单系统只返回比这个 id 新的数据，这样就是避免重复拉取，把漏了的也给补回来了。
第三、不管是用户行为数据，还是销售数据，每天凌晨都会做一个全量校验，用户行为数据对比前端埋点的原始数据，销售数据对比订单系统的全量订单数据，如果发现这边少了，就做一个自动触发补偿的机制，重新去拉取缺失的数据，当然还会生成校验报告，第二天一早来看，确保有没有这种遗漏的情况，基本上是这样。

2.4 数据一致性问题
如何保证用户行为和销售数据的一致性？
这个系统要采集用户的行为和销售数据，这俩数据来自不同的地方，比如用户行为数据是前端埋点，销售数据是订单系统，怎么保证这俩数据的一致性？比如用户下单了，销售数据进来了，但是用户的下单行为数据没进来，这不就数据不一致的问题了吗？这个你怎么解决？


最终一致性方案

销售数据和用户行为数据采集后，分别发送消息到消息队列。

2.5 消息队列可靠性
消息队列若丢消息或消费失败，临时表数据堆积，运营查数据不全怎么办？

消息队列的可靠性与高并发处理” （高级方案）
- 防重复： Redis 缓存 ID + 数据库的 ON DUPLICATE KEY UPDATE (幂等性)。
- 防拥堵： 消费者端限流熔断降级 + 消息积压在 MQ。
- 防重试风暴： 阶梯延迟重试（指数退避） + 随机抖动（Jitter）。
- 死信兜底： 阈值告警降噪 + 凌晨低谷期脚本批量重推。

强化方案可靠性
消息队列采用持久化，生产者确认消息发送成功后返回。
消费端手动 ACK，失败重试三次，仍失败则存入死信队列并告警。
会出现问题：死信队列消息堆积过多   智能重试（指数退避+随机抖动）和降级限流
  - 为了避免把下游数据库彻底打死，我们在 MQ 消费端会配置并发消费线程数的上限（比如 consumeThreadMax），并结合 Sentinel 监控下游数据库的响应时间（RT）。如果发现数据库 RT 突然飙升，比如平时 10ms 涨到了 500ms，Sentinel 会触发熔断降级。这时候消费端会主动暂停拉取新消息，或者放慢消费速度，让消息暂时积压在 MQ 的 Broker 里（MQ 抗积压能力极强，存几千万条都没问题），等数据库恢复了再去消费。
临时表定时任务每小时校验，超过一小时未匹配数据自动触发补偿机制。
每日凌晨全量校验，确保数据准确。

2.6 幂等性保证
前端埋点发送用户行为数据时，如何避免重复发送导致数据重复？

幂等性实现
- 使用唯一键（用户 ID + 行为类型 + 时间戳 + 设备 ID）保证幂等性。
- 数据库写入时用replace into替代insert  ignore into，避免重复插入。
（这个insert ignore into会忽略所有的插入错误。你像比如说这个字段为空了，数据类型不匹配，这些正常的错误也会被忽略的。到时候数据少了，要花大量的时间来排查）
- 接口层加 Redis 请求 ID 校验，设置五分钟过期时间，防止重复请求。
比如说前端发送过来数据的时候，会携带一个比如说请求ID这个东西。我们会把这个请求ID往这个redis当中去存，然后设置一个五分钟的过期时间吧。然后每次接收请求的话，先去查一下redis里面有没有这个请求ID那如果说他有的话，我们就直接返回，那么不做这个后续处理。那没有的话就再继续执行，那么做一个双重保证幂等性。

追问 redis宕机
redis的话，我们肯定在生产环境当中，它不会做单节点的一个部署。比如说我们可以做一个主从架构，就是主节点它宕机之后，重节点它会自动切换，基本上就不会出现这种宕机不可用的情况。然后退一万步来讲的话，就算是真的宕机的，我们还有数据库的一个唯一键来做兜底。Replace in to它还是能保证比如说这个不重复的一个插入，比如说在接口层做校验失败了，那么这个最终数据就是但最终数据它还是不会重复，只不过说是多了一层数据库的一个插入操作。那么可能性能上牺牲了一点点，但是可以保证它不出问题

追问
INSERT ... ON DUPLICATE KEY UPDATE
你这个replace into本质上其实就是delete加insert  并发很高的时候，会不会出现组件冲突？就比如两条相同的这个请求同时到了，然后都过了redis的校验，同时执行replace into会不会报错。
那这个的话我觉得好好解决。就比如说我们在插入操作的时候，在在插入操作的前面，我们可以加一个像分布式锁。比如说用redis的set NX，就是同一时间只有一个请求能执行这个replace in to操作，那么其他的请求去做一个等待，那么执行完就把这个锁给释放掉。那这样子的话就可以避免什么并发冲突。而且锁的这个过去时间我们设置的很短，比如说只有一秒钟，那不会影响什么这个性能对。



2.7 查询性能优化

优化策略
为高频查询条件（用户 ID、商品 ID、时间戳）创建联合索引，避免全表扫描。
对低频查询，使用 Redis 缓存结果，设置过期时间。
数据预聚合，每日凌晨将前一天数据按商品、时间、用户群体聚合，存入聚合表，加速查询。
数据量过大时，按时间范围分表，控制单表数据量。

2.8 大促应急处理

应急方案
优先级处理：保证实时销售和用户行为数据的采集和查询，暂停非核心接口（如选品接口）。
削峰填谷：前端埋点数据先存本地缓存（ConcurrentHashMap），批量写入数据库，避免写入压力过大。
数据补偿：大促后补丢失的数据，更新聚合表。
问题排查：大促结束后，排查服务器资源、数据库压力或消息队列瓶颈，针对性优化。

2.10 要求设计高性能接
设计一个接口，查询某款商品近七天的日销售量趋势，响应时间不超过 500 毫秒，且数据准确。
如果你这个系统要给运营提供一个接口，查询某款商品近七天的是销售量趋势，要求时间不超过500毫秒，而且数据要准确，不能有延迟。你怎么设计这个接口？然后要是遇到查询高峰，比如说100个运营同时查询不同的商品趋势，你怎么你你怎么保证这个接口不超时呢？

接口设计方案
优先查询预聚合表，使用商品 ID 和日期联合索引，保证查询速度。
若当天数据未聚合，查询明细表并合并前六天数据。
- 接口缓存：用 Redis 缓存结果，设置十分钟过期时间，减少数据库压力。
数据库连接池调大，保证高并发查询资源。
- 接口限流：限制每个运营每分钟最多查询十次，避免恶意刷接口。

话术
首先就是我觉得是这样子的，就是我们在设计接口的时候，比如说我会优先去查聚合表。因为近七天的数据我们每天都要聚合。聚合表当中的话它是有比如说商品ID、日期、日销量的这些字段。那么查询的时候，然后我们根据商品ID和日期范围再查聚合表用商品ID加日期的。像这种联合索引，这样子的话就可以保证我们的查询速度很快，基本上能控制在100毫秒之内。
然后如果说为了保证数据的准确的话，我们可以在聚合表当中去加上一个数据更新的时间戳，就是每次聚合任务执行完就更新这个时间戳。那么接口查询的时候去先判断一下这个聚合表的数据它是不是最新的。比如说要是当天的数据聚合表还没有更新，那么就查明细表，把当天的数据和聚合表的前六天数据我们做一个合并，然后再返回给前端。这样子的话就既可以保证这个响应速度，还可以保证这个数据的准确，它没有延迟，对，那基本上是这样子。

追问
100个运营同时查，就算是查聚合表，他也会有并发压力，数据库连接池不够还是会超时，你怎么处理？而且要是运营查的是一款新产品，近七天的数据量很少，聚合表里边也没有查明细表，要是明细表数据量也少。还有就是要是数据量多还好，要是数据量多的话还是会超时的。

首先就是我们在实际的这个场景当中，并没有100个运营同时查，所以说这种情况严格来说确实是不存在的。但是如果说真的是有100个运营同时查的话，那这样子的话他这个并发量并发压力是比较大的。我可以我觉得我们可以先可以去做一个什么叫接口缓存。比如说用redis缓存接口返回的结果，比如说key是商品的ID加时间范围，那么过期时间的话我们给它设置一个十分钟。那这样子的话，运营100个运营同时查同一个商品，那么只有第一个请求会先查数据库，那么后面99个运营他查的话都是查缓存，那么响应时间也会更快，当然对数据库的压力也是有大大的减少。
那另外的话就数据库像这个连接池的话，我们可以把它调大一点。比如说最大的连接数量，我们可以给它调大，然后保证并发查询的时候有足够的连接资源。然后像新品的数据少，我们就就算查明细表，它也不会说出现超时。因为数据量比较少，而且新品的查询频率它本身也不会太高，就算偶尔出现超时的话，也不会影响大部分的运营。另外的话，我们还做了一个接口限流。比如说我们每个运营每十分钟，不是每个运营它每分钟的话最多查十次，就是避免有人恶意刷接口，导致这个服务器压力太大对。

2.3 兜底方案
  - 用户行为数据兜底：
  前端埋点请求失败时，最多重试三次，间隔 1 秒。
  若重试失败，数据存本地缓存（local storage），用户下次打开页面网络恢复后自动重发。
  后端也有兜底，即使前端没有重试
  后端接口接收数据时，先写本地日志记录请求参数和时间。
  若写入数据库失败，定时任务每十分钟扫描日志，对比数据库，缺失则重新写入。
  - 销售数据兜底：
  订单系统同步数据时，通过消息队列（RabbitMQ）发送消息。
  若消息未收到或消费失败，每五分钟定时拉取订单系统接口未同步的数据。
  每日凌晨全量校验，自动补偿缺失数据。

- 追问  
  大促时数据量暴增，定时拉取间隔内丢失数据怎么办？本地日志存储过多会不会占用内存或拖慢服务？
    缩短定时拉取间隔（如五分钟改一分钟），快速补漏。
    本地日志采用滚动日志，保留 24 小时内数据，超过自动删除。
    日志文件按大小切割，单个文件不超过 100MB，避免占用过多空间和拖慢服务。

3. 消息队列与流量削峰 (数据怎么缓冲)
- 真实场景： 大促期间（比如双十一），瞬间涌入平时几十倍的请求量。如果把这些请求直接写入关系型数据库，数据库连接池会被瞬间打满导致系统崩溃。引入 MQ（如 Kafka 或 RocketMQ）相当于建了一个巨大的“蓄水池”，接住瞬间的洪峰，再让下游系统以自己能承受的速度慢慢处理。
- 面试重点： 你们用了哪种 MQ 技术栈？大促时的峰值 QPS（每秒查询率）大概是多少？如何处理极端情况下的 MQ 消息积压？如何保证消息不丢失？
4. 异步落库与主链路解耦 (数据怎么存)
- 真实场景： 消费者程序从 MQ 中拉取数据，清洗后写入最终的存储（如离线数仓 Hive、实时分析 ClickHouse 或业务线 MySQL）。“异步”意味着用户点击“加购”后，主业务直接返回成功，而不必等待数据入库完成，从而大幅降低了接口的响应时间（提升吞吐量）。
- 面试重点： 消费失败了怎么重试？如何保证幂等性（即如果网络抖动导致同一条消息被消费了两次，数据库里如何保证不会多出一条重复记录）？
